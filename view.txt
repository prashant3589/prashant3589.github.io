Story

As a DevOps/Backend Engineer,
I want to standardize the process of converting Entity Framework (EF) migrations into Flyway-compatible scripts and executing them,
so that database changes are consistent, version-controlled, and reliably deployed across environments.

Background / Context

Currently, EF migrations are used for schema changes, but there is no standardized approach to:

Convert EF migrations into SQL scripts compatible with Flyway
Maintain versioning and naming conventions
Execute migrations consistently across environments

This leads to:

Inconsistent deployments
Manual errors
Lack of auditability
Scope
Define approach for:
Generating SQL scripts from EF migrations
Transforming scripts into Flyway format
Versioning and naming conventions
Execution of Flyway migrations via pipeline
Cover environments: Dev, QA, Prod
Acceptance Criteria
✅ 1. EF to SQL Script Generation

A standard command/process is defined:

dotnet ef migrations script <from> <to> -o <output.sql>
Supports:
Incremental migration
Full baseline script
✅ 2. Flyway Script Formatting

Scripts follow Flyway naming convention:

V<version>__<description>.sql

Example:

V1_0_0__create_employee_table.sql
Script must:
Be idempotent (if required)
Avoid EF-specific metadata where unnecessary
Be reviewed before commit
✅ 3. Baseline Strategy

Initial baseline defined using:

flyway baseline
Baseline version documented (e.g., V1.0.0)
Existing DB state aligned with baseline
✅ 4. Script Storage

All scripts stored in:

/database/flyway/migrations
Version-controlled via Git
✅ 5. Flyway Execution
Flyway execution integrated in pipeline:
Validate
Migrate

Command example:

flyway migrate
