Slide 1 — Title
Entity Framework + Flyway Database DevOps Demo

Automating SQL Server schema changes using EF Migrations, Flyway, and Azure DevOps

Objective:
Enable governed, repeatable, auditable database deployments through CI/CD pipelines.

Slide 2 — What is Flyway?
Flyway Overview

Flyway is a database migration tool used to version, track, validate, and deploy schema changes across environments.

Core Capabilities
Versioned SQL migrations (V1__Init.sql, V2__AddTable.sql)
Ordered deployment execution
Change history tracking
Validation & checksum verification
Drift detection
Repeatable migrations
Rollback strategy support (process-based)
CI/CD integration
Why Use Flyway?
Controlled production deployments
Source-controlled database changes
Auditability & compliance
Prevents manual script execution risks
Slide 3 — Why Flyway with Entity Framework?
Challenge with EF-Only Approach

Entity Framework migrations are developer-friendly but often lack enterprise deployment governance.

Combined Model

Use EF for:

Developer productivity
Model-first schema generation

Use Flyway for:

SQL artifact versioning
Controlled promotion to environments
Pipeline-based execution
Drift / validation reporting
Best of Both Worlds
Developer Change → EF Migration → SQL Script → Flyway Deployment

Slide 7 — Developer Workflow (EF to Flyway)
Step-by-Step
Create feature branch from develop
Modify model/entities
Generate EF migration
dotnet ef migrations add AddCustomerTable
Generate SQL script from EF migration
dotnet ef migrations script
Convert output into Flyway versioned script
V20260421_01__AddCustomerTable.sql
Commit:
Application code
EF migration
Flyway script
Raise PR
Slide 8 — Suggested Versioning Strategy
Recommended Naming
V20260421_1530__AddCustomerTable.sql
Why Timestamp Versions?
Avoid collisions between developers
Supports parallel development
Easier sequencing
Avoid
V5.sql / V6.sql

in high-concurrency teams.
